Documentacion detallada por archivo

Este documento explica cada archivo importante del proyecto CRUD de empleados con React Native y Expo. La idea es que un estudiante pueda entender para que sirve cada archivo, que responsabilidad tiene y como se conecta con los demas.

1. Alcance de esta documentacion

Esta documentacion cubre los archivos propios del proyecto. No se documenta la carpeta node_modules porque contiene dependencias externas instaladas automaticamente. Tampoco se explica archivo por archivo el contenido de package-lock.json, ya que es un archivo generado por npm para fijar versiones.

Archivos funcionales documentados: App.js, package.json, app.json, babel.config.js, README.md y todos los archivos dentro de src.

Entrada principal

App.js

App.js

Este es el punto de entrada visual de la aplicacion. Su responsabilidad principal es configurar la navegacion entre pantallas y definir el estilo comun del encabezado.

Que importa

Importa React, el contenedor de navegacion, el stack de React Navigation, el StatusBar y las tres pantallas principales.

Que crea

Crea una instancia de createNativeStackNavigator() para manejar la navegacion tipo pila.

Que registra

Registra tres rutas: Lista, Crear y Editar.

Por que es importante

Conecta toda la app. Sin este archivo, las pantallas existirian, pero no habria forma de navegar entre ellas.

Tambien define un estilo comun para el encabezado: fondo azul, texto blanco y fondo claro en el contenido. Esto hace que la aplicacion se vea coherente sin repetir estilos en cada pantalla.

Bloqueo de versiones

package-lock.json

package-lock.json

Este archivo es generado automaticamente por npm. Su funcion es guardar el arbol exacto de dependencias que se instalo en el proyecto.

Aunque normalmente no se edita manualmente, es importante porque ayuda a que varias personas del equipo instalen exactamente las mismas versiones, reduciendo errores de "en mi maquina si funciona".

En una explicacion para estudiantes, se puede presentar como un archivo de apoyo tecnico: no contiene logica de negocio de la app, pero si asegura estabilidad en las instalaciones.

Capa de datos

src/services/api.js

src/services/api.js

Este archivo centraliza toda la comunicacion con la API REST de empleados. Es uno de los archivos mas importantes porque separa la logica de red de la logica visual.

BASE_URL

Guarda la ruta base https://nodejs-test-tfzg.onrender.com/api/empleados. Si cambia la API, este es el primer lugar a revisar.

request()

Funcion reutilizable que ejecuta fetch, aplica cabeceras JSON, interpreta la respuesta y maneja errores.

normalizeListResponse()

Permite soportar respuestas en forma de arreglo directo o en forma de objeto con propiedad data.

normalizeEmployeeResponse()

Hace una normalizacion similar pero para respuestas de un solo empleado.

Funciones exportadas y su responsabilidad:

  • getEmployees(): obtiene la lista completa de empleados.
  • getEmployeeById(id): obtiene un empleado especifico por su identificador.
  • createEmployee(employee): envia un POST para crear un empleado.
  • updateEmployee(id, employee): envia un PUT para actualizar un empleado.
  • deleteEmployee(id): envia un DELETE para eliminar un empleado.

Pedagogicamente, este archivo enseńa una idea clave: la interfaz visual no deberia construir URLs ni manejar detalles HTTP por todas partes. Es mejor concentrar eso en un unico servicio.

Pantalla principal

src/screens/EmployeeListScreen.js

src/screens/EmployeeListScreen.js

Esta pantalla muestra el listado de empleados. Es la primera vista que el usuario encuentra al abrir la app.

Estados usados

Maneja employees, loading, refreshing y message.

loadEmployees()

Consulta la API usando getEmployees() y actualiza la lista local.

useFocusEffect

Hace que la lista se recargue cuando la pantalla vuelve a estar activa, por ejemplo despues de crear o editar.

handleDelete()

Elimina en la API y luego filtra el arreglo local para reflejar el cambio sin esperar otra consulta completa.

En la interfaz usa estos elementos importantes:

  • MessageBox para mostrar mensajes de exito o error.
  • Un boton Pressable para ir a la pantalla de crear.
  • FlatList para renderizar cada empleado.
  • RefreshControl para permitir refrescar con gesto de arrastre.
  • EmployeeItem como componente de cada tarjeta del listado.

Esta pantalla es un gran ejemplo para estudiantes porque concentra varios conceptos comunes: carga inicial, estados, consumo de API, renderizado de listas, navegacion y actualizacion visual despues de eliminar.

Pantalla de creacion

src/screens/CreateEmployeeScreen.js

src/screens/CreateEmployeeScreen.js

Esta pantalla tiene una responsabilidad muy concreta: crear un empleado nuevo.

Su diseńo es deliberadamente sencillo. No construye un formulario propio, sino que reutiliza EmployeeForm. Eso ayuda a enseńar el principio de reutilizacion de componentes.

Cuando el usuario guarda, la funcion handleCreate() llama a createEmployee() del servicio API. Si todo sale bien, navega de vuelta a la pantalla Lista y envia un mensaje de exito por parametros.

Es un buen ejemplo de una pantalla delgada: contiene poca logica porque delega el formulario al componente y la peticion HTTP al servicio.

Pantalla de edicion

src/screens/EditEmployeeScreen.js

src/screens/EditEmployeeScreen.js

Esta pantalla se encarga de editar un empleado existente. Es ligeramente mas compleja que la pantalla de crear, porque puede necesitar cargar el detalle del empleado antes de mostrar el formulario.

Datos que recibe

Recibe desde la navegacion employeeId y, en algunos casos, un objeto employee ya disponible.

useEffect

Si no llego un empleado completo desde la lista, consulta la API con getEmployeeById().

loading y error

Muestra un indicador de carga o un mensaje de error antes de renderizar el formulario.

handleUpdate()

Envía los cambios con updateEmployee() y luego vuelve a la lista con un mensaje de confirmacion.

El codigo usa la variable isMounted para evitar actualizar el estado cuando el componente ya no esta montado. Esto ayuda a prevenir errores comunes en operaciones asincronas.

Componente reutilizable

src/components/EmployeeForm.js

src/components/EmployeeForm.js

Este componente representa el formulario de empleados. Se reutiliza tanto para crear como para editar, por lo que es una pieza central del proyecto.

Props que recibe:

  • initialValues: valores iniciales del formulario. En crear llegan vacios y en editar llegan con datos existentes.
  • buttonText: texto del boton principal.
  • onSubmit: funcion que se ejecuta al guardar.

Estados

Guarda cada campo del formulario por separado: name, position, office, salary, ademas de loading y mensajes.

Validaciones

Valida que todos los campos esten diligenciados y que el salario sea numerico.

handleSave()

Construye el objeto final del empleado, convierte el salario a numero y llama a onSubmit.

Retroalimentacion

Usa MessageBox y ActivityIndicator para informar al usuario lo que esta pasando.

Desde una mirada pedagogica, este archivo enseńa formularios controlados, validacion sencilla, conversion de tipos y reutilizacion de componentes.

Tarjeta de listado

src/components/EmployeeItem.js

src/components/EmployeeItem.js

Este componente representa un empleado dentro de la lista. Su trabajo es puramente visual e interactivo: mostrar la informacion y ofrecer acciones de editar y eliminar.

Props que recibe:

  • employee: objeto con los datos del empleado.
  • onEdit: funcion que se ejecuta al pulsar el boton Editar.
  • onDelete: funcion que se ejecuta al confirmar la eliminacion.

Antes de eliminar, usa Alert.alert() para pedir confirmacion. Esto es importante porque evita borrar por accidente.

Visualmente, muestra nombre, cargo, oficina y salario. Luego renderiza dos botones con colores distintos para dejar clara la accion: azul para editar y rojo para eliminar.

Mensajes reutilizables

src/components/MessageBox.js

src/components/MessageBox.js

Este componente muestra mensajes simples de exito o error. Su logica es intencionalmente minima.

type

Define si el mensaje es success o error.

message

Es el texto que se va a mostrar al usuario.

Aunque es pequeno, este archivo enseńa una idea muy util: convertir un patron visual repetido en un componente reutilizable.

9. Relacion entre archivos

El flujo general del proyecto se puede entender asi:

  • App.js registra las pantallas y controla la navegacion.
  • EmployeeListScreen.js muestra el listado inicial y usa EmployeeItem para cada registro.
  • CreateEmployeeScreen.js y EditEmployeeScreen.js reutilizan EmployeeForm.
  • EmployeeForm.js recoge datos del usuario y llama a una funcion que recibe por props.
  • Las pantallas llaman al servicio en src/services/api.js para hablar con la API real.
  • MessageBox.js se usa para retroalimentar al usuario con mensajes de estado.
Una buena manera de enseńarlo en clase es seguir esta ruta: App.js -> pantallas -> componentes -> servicio API. Ese recorrido ayuda a los estudiantes a entender primero la interfaz y luego la comunicacion con el servidor.
